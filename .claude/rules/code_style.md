# Стиль кода и соглашения

> **Правило самообновления:** Обновляйте этот файл при изменении правил линтинга или соглашений по коду в `analysis_options.yaml` или командных договорённостях.

## Форматирование
- Ширина строки: **100** символов
- Одинарные кавычки для строк
- Trailing commas обязательны (в списках параметров, коллекциях и т.д.)
- Перевод строки в конце файла

## Именование
- Файлы: `snake_case` (правило `file_names`)
- Классы, перечисления, расширения, typedef'ы: `PascalCase`
- Константы: `lowerCamelCase`
- Без ведущих подчёркиваний для локальных идентификаторов

## Импорты
- **Относительные импорты** предпочтительнее package-импортов (`prefer_relative_imports: true`)
- Сортировка импортов обязательна (`directives_ordering: true`)

## Иммутабельность
- Предпочитать `const`-конструкторы везде, где возможно
- Предпочитать `final` для локальных переменных, итераторов for-each и полей классов
- Предпочитать `const`-литералы в конструкторах иммутабельных классов

## Null Safety
- Избегать явной инициализации nullable-переменных значением `null`
- Использовать операторы `??`, `??=`, `?.` вместо ручных проверок на null
- Избегать приведения nullable к non-nullable через `as`

## Специфика Flutter
- Все виджеты должны объявлять параметр `key` (`use_key_in_widget_constructors`)
- `createState()` должен только возвращать экземпляр state — без логики
- Избегать лишних виджетов `Container`
- Отменять подписки и закрывать sink'и при dispose
- Использовать `BuildContext` синхронно в async-методах
- Не использовать `void` как возвращаемый тип для async-функций — использовать `Future<void>`

## Управление потоком
- Использовать `for` вместо `.forEach()` на коллекциях
- Предпочитать `if`-элементы в коллекциях вместо условных выражений
- Исчерпывающий `switch/case` — избегать `default`-кейсов
- Без управления потоком (`return`, `break`) в блоках `finally`

## Общие правила
- Без `print()` — использовать логгер Talker
- Без `dynamic`-вызовов (`avoid_dynamic_calls: true`)
- Бросать только подклассы `Error` (`only_throw_errors`)
- Предпочитать expression function bodies (`=>`) где возможно
- Предпочитать строковую интерполяцию вместо конкатенации
- Предпочитать каскадную нотацию для нескольких операций над одним объектом
- `assert` должен содержать сообщение; предпочитать assert'ы в initializer lists
- `unawaited_futures` — всегда использовать `await` или оборачивать в `unawaited()`

## Кодогенерация
- Никогда не редактировать вручную файлы `*.freezed.dart`, `*.g.dart`, `*.config.dart`
- Выполнять `make generate` после изменения аннотированных классов
- Сгенерированные файлы исключены из анализа
